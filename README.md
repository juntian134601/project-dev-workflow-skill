# Project Development Workflow

这是一个可同时用于 Codex 和 Claude 的项目开发 Skill。优先通过 `/npp` 触发，用结构化流程把“想改什么”推进到“分析、规划、增量实现、验证”。

## 触发方式

优先触发关键词：

```text
/npp <你想要修改或开发的方向>
```

示例：

```text
/npp 给后台管理系统增加 RBAC 权限控制
/npp 修复登录后刷新页面状态丢失的问题
/npp 分析这个项目并规划一个会员订阅功能
```

当用户输入以 `/npp` 开头时：

1. 将 `/npp` 后面的内容视为本次开发目标。
2. 先判断任务类型：新项目、新功能、现有功能修改、bug 修复、重构规划、项目分析。
3. 根据任务复杂度选择完整流程或 lite 流程。
4. 除非用户明确要求“只分析”或“只写计划”，否则在完成必要分析和计划后进入增量实现。

兼容触发表达：

- `帮我开发这个项目`
- `开始一个新功能`
- `按流程开发`
- `分析项目并编写 spec`
- `AI 开发流程`
- `结构化开发`

## 流程选择

### 完整流程

用于新项目、复杂功能、大范围改造、权限/支付/数据模型/架构相关任务。

```text
Step1 理解需求 -> Step2 任务分解 -> Step3 分析项目 -> Step4 编写 Spec
-> Step5 评审 Spec -> Step6 实施计划 -> Step7 增量编码
```

### Lite 流程

用于明确的小改动、bug 修复、文案调整、局部 UI 修改。

```text
Step3-lite 针对性分析 -> Step6-lite 简短计划 -> Step7 增量编码 -> 验证
```

如果任务看似简单但涉及认证、权限、数据删除、支付、外部 API、数据库迁移或跨模块状态，升级为完整流程。

## 7 步工作流程

| 步骤 | 阶段 | 核心动作 | 输出 |
|------|------|----------|------|
| Step1 | 理解需求 | 澄清业务目标、用户、范围、约束和验收标准 | 需求摘要 |
| Step2 | 任务分解 | 拆模块、定边界、排依赖、估复杂度 | 任务清单 |
| Step3 | 项目分析 | 只读分析技术栈、目录结构、状态管理、API、UI、权限、路由、构建和代码规范 | 项目分析摘要 |
| Step4 | 编写 Spec | 描述功能目标、用户角色、页面结构、数据结构、API、权限、风险和验收标准 | Spec 文档 |
| Step5 | 评审 Spec | 对齐边界、发现冲突、确认高风险决策 | 确认项或待确认问题 |
| Step6 | 实施计划 | 分阶段、列文件、标风险、标影响范围、定义验证方式 | 实施计划 |
| Step7 | 增量编码 | 按阶段实施，保持小批量修改，完成后验证 | 代码变更和验证结果 |

## 执行规则

1. 先读取项目上下文，再做设计或改动。
2. 高风险或需求不清时先向用户确认；目标明确的小改动可以直接执行。
3. 优先小批量修改；如果一次需要改超过 5 个文件，先说明原因和影响范围。
4. 遵循项目现有技术栈、目录结构、命名方式、组件风格和测试习惯。
5. 不做无关重构，不修改无关模块，不覆盖用户已有改动。
6. 实现后运行与改动范围匹配的测试、构建或手动验证。
7. 最终回复说明修改内容、涉及文件、验证结果和剩余风险。

## Step3-lite

在 bug 修复或小改动中使用 Step3-lite：

1. 只读取与问题直接相关的文件、调用链、配置和测试。
2. 找到至少一个相似实现或现有模式。
3. 说明判断依据，然后进入简短计划或直接修复。

## Spec 要求

编写 Spec 时包含：

- 目标和非目标
- 用户角色和关键场景
- 页面、接口、数据或状态变化
- 权限和边界条件
- 风险点和回滚思路
- 验收标准

## 跨平台使用说明

在 Codex 中使用时，将本文件放入 skill 目录并命名为 `SKILL.md`。

在 Claude 中使用时，可以把本文件作为 Claude Skill 的 `SKILL.md`，也可以把 `/npp` 配置成 slash command，让命令内容转交给此流程处理。无论平台是否真正支持自动 slash command，只要用户消息以 `/npp` 开头，都按本 Skill 的触发方式处理。
